# Supply Chain Security

Every image pushed to GHCR on `main` goes through three supply chain security steps.

## SBOM

A Software Bill of Materials is generated by [Trivy](https://trivy.dev/) from the
runtime image in **SPDX JSON** format and is available in two places:

- **Workflow artifact** — downloadable from the GitHub Actions run summary, retained for 90 days
- **Registry attestation** — attached to the image digest via Cosign (see below)

## Cosign signing

Images are signed using [Cosign](https://docs.sigstore.dev/cosign/overview/) with
**keyless signing** — no private key is stored anywhere. The signature is produced
using the GitHub Actions OIDC token and recorded in the
[Sigstore transparency log (Rekor)](https://rekor.sigstore.dev/).

The signature is stored in GHCR alongside the image as an OCI artifact.

### Verify an image

```bash
cosign verify \
  --certificate-identity-regexp="https://github.com/<owner>/python-zero-code-demo/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  ghcr.io/<owner>/python-zero-code-demo:<tag>
```

### Verify the SBOM attestation

```bash
cosign verify-attestation \
  --type spdxjson \
  --certificate-identity-regexp="https://github.com/<owner>/python-zero-code-demo/.*" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  ghcr.io/<owner>/python-zero-code-demo:<tag>
```

## Provenance

The `docker/build-push-action` generates a signed **SLSA provenance** attestation
(`provenance: true`) and an **OCI SBOM** (`sbom: true`) for every pushed image,
stored as attestations in GHCR.

## Vulnerability scanning

[Trivy](https://trivy.dev/) scans the runtime image on every CI run. The pipeline
fails on any **unfixed** `CRITICAL` or `HIGH` severity vulnerability.

```
Severity filter : CRITICAL, HIGH
Unfixed only    : yes (ignore vulnerabilities with no available fix)
Exit code       : 1 (blocks the pipeline)
```
